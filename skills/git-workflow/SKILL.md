# SKILL: git-workflow

## Purpose

This skill handles all Git operations for the Enterprise AI Workflow: creating feature branches from the latest `main`, staging and committing scaffolded files, pushing to GitHub, and reporting the result back to Slack. It is always the final execution step after `angular-scaffold` completes.

**Only activate after `angular-scaffold` (or `devops-tasks`) has finished generating files.**

---

## When to Activate

- After `angular-scaffold` finishes generating all files
- After `devops-tasks` finishes making infrastructure changes
- When Kiet explicitly asks to commit and push current changes

---

## Step-by-Step Procedure

### Step 0 — Surgical Stage Check

Before touching git, re-run the Surgical checkpoint from [.agent/GUIDELINES.md §3](../../.agent/GUIDELINES.md#3-surgical-changes):

- `git status` output MUST contain only files produced by this task. Anything unrelated → halt, ask Kiet.
- `git diff` MUST show no incidental reformatting or "drive-by" edits.
- Any pre-existing dead code you noticed during scaffold → mention in Slack, do NOT delete in this commit.
- The commit message MUST trace to the approved plan (title, scope, refs).

If any check fails → halt, report in Slack, do not stage.

See [skills/guideline/SKILL.md](../guideline/SKILL.md) §Checkpoint 3 for the full question set.

---

### Step 1 — Verify Clean Starting Point

Before creating a branch, confirm the working state:

```bash
cd [project-directory]

# Check current branch
git branch --show-current

# Check for any uncommitted changes from before this task
git status

# Check latest commit on main
git log main --oneline -5
```

If there are pre-existing uncommitted changes NOT from this task, **stop and ask Kiet** before proceeding.

---

### Step 2 — Sync with Latest Main

Always branch from the latest `main` to avoid conflicts:

```bash
git checkout main
git pull origin main

# Confirm we're up to date
git log --oneline -3
```

If `git pull` results in conflicts, **stop and report to Slack** — do not attempt to resolve merge conflicts automatically.

---

### Step 3 — Create Feature Branch

Branch naming follows this exact format: `feature/[task-id]-[kebab-description]`

```bash
# Format: feature/[task-id]-[short-kebab-description]
# Examples:
#   feature/FE-123-user-list-page
#   feature/FE-456-add-payment-form
#   feature/INFRA-12-update-nginx-config

git checkout -b feature/[task-id]-[description]

# Confirm branch was created
git branch --show-current
```

**Branch naming rules:**

- Always prefix with `feature/`
- Use the Jira issue key if available (e.g., `FE-123`)
- Use `task-[date]` if no issue key (e.g., `task-20250419`)
- Description in kebab-case, max 5 words
- No uppercase, no spaces, no special characters except `-`

---

### Step 4 — Stage Files

Stage only the files generated by this task. Never stage unrelated files.

```bash
# Stage specific files from the plan
git add src/app/features/[feature]/
git add src/app/shared/[new-files-if-any]

# If routing was modified
git add src/app/[routing-file]

# Review what will be committed
git status
git diff --cached --stat
```

**Never use `git add .` blindly.** Always review `git status` first to confirm only task-related files are staged.

---

### Step 5 — Commit

Use Conventional Commits format. One commit per task (unless the plan explicitly requires multiple).

```bash
git commit -m "[type]([scope]): [description]

[optional body — list of files and what they do]

[optional footer — Jira issue reference]"
```

**Commit types:**

| Type       | When to Use                              |
| ---------- | ---------------------------------------- |
| `feat`     | New feature or component                 |
| `fix`      | Bug fix                                  |
| `refactor` | Code restructure without behavior change |
| `style`    | Formatting, CSS changes                  |
| `chore`    | Config, tooling, dependencies            |
| `docs`     | Documentation only                       |
| `test`     | Adding or updating tests                 |
| `ci`       | CI/CD pipeline changes                   |
| `infra`    | Infrastructure/DevOps changes            |

**Examples:**

```bash
# Feature commit
git commit -m "feat(user-list): scaffold smart/dumb components and service

- UserListComponent (smart): container with data stream
- UserListUiComponent (dumb): presentational with OnPush
- UserListService: HTTP CRUD operations
- UserModel: typed interface

Refs: FE-123"

# DevOps commit
git commit -m "infra(nginx): update proxy config for new API gateway

- Updated upstream server address
- Added health check endpoint
- Increased timeout to 60s

Refs: INFRA-12"
```

---

### Step 6 — Push to GitHub

```bash
git push origin feature/[task-id]-[description]

# Confirm push was successful
git log --oneline -3
echo "Remote branches:"
git branch -r | grep feature/[task-id]
```

If push fails due to authentication, check that `GITHUB_TOKEN` is set in the environment.

---

### Step 7 — Create Pull Request (optional)

Create a PR only if Kiet's plan or approval message requests it. Otherwise, just push the branch.

```bash
gh pr create \
  --title "feat([scope]): [description]" \
  --body "$(cat plan.md)" \
  --base main \
  --head feature/[task-id]-[description] \
  --label "ai-scaffold"
```

If `plan.md` doesn't exist, use this body template:

```bash
gh pr create \
  --title "feat([scope]): [description]" \
  --body "## Summary
Scaffolded by ZenAgent

## Changes
- [list files created]
- [list files modified]

## Notes
Business logic implementation pending — see plan for details.

Refs: [Jira issue key]" \
  --base main \
  --head feature/[task-id]-[description]
```

---

### Step 8 — Clean Up

After successful push:

```bash
# Remove temporary plan.md from project root (if it was created there)
rm -f plan.md

# Return to main branch
git checkout main
```

---

### Step 9 — Report to Slack

Send a completion report to Slack in the same thread as the task:


## Error Handling

| Situation                        | Action                                                                                                      |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `git pull` has conflicts         | Stop. Report to Slack: "Conflicts on main — please resolve before I proceed."                               |
| Push rejected (non-fast-forward) | Stop. Report to Slack: "Push rejected — branch may already exist with different history."                   |
| Authentication failure           | Check `GITHUB_TOKEN` env var. Report to Slack if unresolvable.                                              |
| Pre-existing uncommitted changes | Stop. Ask Kiet: "There are uncommitted changes in the project. Should I stash them or is this intentional?" |
| Wrong branch created             | `git branch -d [wrong-branch]` then retry Step 3.                                                           |
